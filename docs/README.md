# OctoAcme Project Management Docs

## Overview

This README is the central entry point to OctoAcme's project management process documentation. It summarizes our approach and links to each detailed process guide stored in this folder.

## Project Management Practices Summary

OctoAcme manages projects through a structured, end-to-end lifecycle composed of five phases: **Initiation**, **Planning**, **Execution & Tracking**, **Release**, and **Close & Retrospective**. Work begins in Initiation, where a **Project One-pager** captures the problem statement, SMART objective, success metrics, primary stakeholders, high-level timeline, initial risks, and resource needs. A decision gate — requiring clear success metrics, confirmed stakeholder priority, and available team capacity — must be cleared before moving to Planning. In Planning, the approved initiative is converted into a prioritized, estimated backlog of shippable increments, each with explicit **acceptance criteria** and a **Definition of Done (DoD)**, alongside a release timeline and milestone map. Risks and cross-team dependencies are captured in a **Risk Register** (ID, description, impact, likelihood, owner, mitigation, and status) from this phase onward and are continuously monitored through weekly syncs.

Day-to-day **Execution & Tracking** is built around strong visibility and iterative delivery. Teams maintain a project board (e.g., GitHub Projects) with a standard column flow of **Backlog → Ready → In Progress → In Review → QA → Done** and a steady meeting rhythm: **daily standups** (15 min, focused on progress, blockers, and dependencies), a **weekly delivery sync** (progress updates and flagged risks), and **demos or reviews** at the end of each sprint or milestone. The **PR workflow** reinforces quality at the code level: PRs are kept small (≤ 400 lines where possible), must link the related issue and include acceptance criteria in the description, require CI checks (automated tests and linting) to pass before review, and need at least one approval before merging. When issues cannot be resolved at the team level, a clear **escalation path** applies — team triage in the daily standup → PM escalates to Product Lead and dependent teams → sponsor-level escalation for business-impacting issues. After each release or sprint, a **blameless retrospective** captures what went well, what could be improved, and time-bound action items (with named owners) that are tracked back into the project backlog for continuous improvement.

OctoAcme's five **core roles** each have distinct responsibilities to ensure clear ownership and accountability. The **Project Manager (PM)** coordinates delivery, manages schedules, risks, and dependencies, facilitates meetings, maintains project documentation and status reporting, and leads cross-team and stakeholder communication. The **Product Manager / Product Lead (PdM)** defines problem statements and success metrics, prioritizes the roadmap and backlog, collaborates with engineering and stakeholders on trade-offs, and validates outcomes through user research and data. **Developers** implement features and fixes to meet acceptance criteria, write and maintain tests and documentation, participate in design and code reviews, assist in estimation and planning, and surface technical risks. **QA/Testing** validates quality and acceptance criteria, while **Stakeholders** provide inputs and approvals. These roles are supported by a shared set of key artifacts — the Project One-pager, roadmap and release plan, sprint backlog, DoD, risk register, and retrospective notes — that serve as a single source of truth across the project.

Communication and quality assurance are treated as continuous, structured practices rather than end-stage activities. Stakeholder communication follows a defined cadence: a **weekly PM + PdM alignment**, **twice-weekly (or as-agreed) standups** for the delivery team, and **monthly stakeholder updates**; a standard **Weekly Status Template** (progress this week, next steps, risks & blockers, asks/decisions needed) keeps updates consistent. A **single source of truth** (project README or release doc) is maintained so all stakeholders can find current status without ambiguity. Quality is enforced throughout: unit and integration tests cover new logic, **CI pipelines** run automated tests, linting, and **security scanning** before any merge, and **end-to-end smoke tests** are run against critical flows before every release. Releases follow a standardized **deployment checklist** — acceptance criteria met, CI and security scans passing, release notes drafted, rollback/mitigation plan documented, staging smoke tests completed, post-deploy verification run, and release announced to stakeholders and support. A post-incident **blameless retrospective** is also used after any production incident, producing owned, time-bound action items that flow back into the backlog, reinforcing a culture of continuous improvement.
